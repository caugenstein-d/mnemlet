# Mnémlet v0.3 Publication-Grade Trust / Security / Privacy Design

**Date:** 2026-05-26  
**Status:** Approved design target; implementation plan pending  
**Project:** Mnémlet  
**Release target:** `v0.3.0` as the public-release gate after v0.2 polish  
**Prior spec:** `docs/superpowers/specs/2026-05-22-trust-security-privacy-layer-design.md`

---

## 1. Goal

Mnémlet v0.3 is the Trust / Security / Privacy release. Its goal is not merely to add features. Its goal is to make Mnémlet publicly defensible for self-hosted use.

After v0.3, the project should be ready for a public release path: updated docs, updated demo, local daemon deployment, tag decision, and then a separate PyPI/publication step only after final approval.

The public criticism v0.3 must answer is simple: "This memory service has no auth, no secret guard, no audit trail, and no recovery story." v0.3 closes that gap without turning Mnémlet into an enterprise product.

---

## 2. Scope

v0.3 includes the P0 and P1 feature sets from the prior Trust / Security / Privacy draft.

### P0 — required

- Single API Key Auth.
- Secret Regex Guard.
- Audit Log.
- Startup Security Check.

### P1 — included in v0.3

- Namespace Policies.
- Explain+ trust extension.
- Forget / Replace / Confirm trust extensions.
- Backup / Restore full snapshot.

### Out of scope

- RBAC or multi-user support.
- Per-agent API keys.
- TLS termination inside Mnémlet.
- Encryption at rest inside Mnémlet.
- LLM-based secret detection.
- Selective export.
- Audit retention policies.
- Public publishing or PyPI upload during implementation.

These omissions are deliberate. Mnémlet remains a solo, local-first, self-hosted memory engine. OS-level security, reverse proxies, and full multi-user isolation stay outside this release.

## 3. Release-quality bar

v0.3 is successful only if it is good enough to publish. Passing tests alone is not sufficient.

### Technical bar

- `.venv/bin/pytest -q` passes.
- Security-specific tests cover REST and MCP auth.
- Secret Guard tests cover `block`, `warn`, and `allow`.
- Audit tests cover write, query, denial, block, warning, and internal/system events.
- Startup checks are tested for unsafe host, missing key, and unsafe file permissions.
- Explain+ returns trust fields and audit trail without breaking existing provenance.
- Namespace Policies are tested for soft warnings and `confirm_before_forget`.
- Backup / Restore roundtrip is tested with isolated data directories.
- Existing v0.2 APIs stay compatible unless an API key is explicitly configured.

### Security bar

- API keys are never logged, stored in cleartext, included in backups, or returned by APIs.
- Audit entries never store full request bodies.
- Secret Guard error messages disclose pattern types, not the full secret value.
- Auth failures do not leak useful information beyond `401 Unauthorized`.
- Startup warnings make unsafe configuration obvious.
- README no longer says there is no auth layer once v0.3 lands.

### Product and documentation bar

- README Quickstart explains local warning mode and recommended API-key mode.
- REST examples include auth headers where appropriate.
- MCP / OpenWebUI / OpenCode examples show token configuration.
- Website surfaces v0.3 Trust / Security / Privacy truth.
- Demo shows auth, Secret Guard block, Audit, and Explain+.
- SECURITY.md and CONTRIBUTING.md match v0.3 behavior.
- Changelog / release notes are prepared before public publication.

## 4. Architecture

v0.3 adds a Trust layer. It must not become scattered route-local `if` statements.

```text
src/mnemlet/
  security/
    __init__.py
    auth.py               # API key config, hashing, request validation
    audit.py              # Audit event model, writer, query helpers
    secret_guard.py       # Regex scanner and namespace action resolution
    startup_check.py      # Unsafe config and permission warnings
    namespace_policies.py # Soft policies and confirm_before_forget

  backup/
    __init__.py
    backup.py             # Snapshot creation
    restore.py            # Restore flow and pre-restore backup
```

### Integration points

- **FastAPI middleware:** Auth and request-level audit. These are cross-cutting and should not be optional per-route by accident.
- **Route / engine hooks:** Secret Guard and Namespace Policies. These apply only where content or policy decisions exist.
- **Storage layer:** Additive migrations for `audit_log`, Trust fields on `memories`, and `namespace_policies`.
- **MCP:** Same security model as REST. No second, weaker path.
- **CLI:** Key generation, backup, and restore commands.

Auth and Audit are middleware because they apply to every request. Secret Guard and Namespace Policies are hooks because they are operation-specific.

## 5. Security behavior

### Auth

Configuration sources:

```toml
[auth]
api_key = "mnemlet_..."
```

or:

```bash
MNEMLET_API_KEY=mnemlet_...
```

Request formats:

- `X-Mnemlet-Key: <key>`
- `Authorization: Bearer <key>`

Behavior:

| State | Behavior |
|---|---|
| No API key configured | Requests are allowed; startup and status warn; audit marks caller as `unauthenticated`. |
| API key configured and valid | Request is allowed. |
| API key configured and missing / wrong | Request returns `401 Unauthorized`; denial is audited without storing the key. |

This keeps existing local setups working until a key is intentionally configured.

### Secret Guard

Default action is `block`.

Required standard detections:

- GitHub PATs like `ghp_...`.
- OpenAI-style `sk-...` keys.
- AWS Access Key IDs like `AKIA...`.
- Generic long hex/API-key contexts.
- Password assignments like `password=...`, `passwd: ...`, or `pwd=...`.
- Optional email detection through namespace policy, not globally hard-blocked.

Actions:

| Action | Result |
|---|---|
| `block` | Write is rejected; no memory is stored; audit result is `blocked`. |
| `warn` | Write succeeds; memory metadata marks the guard result; audit result is `warning`. |
| `allow` | Scan is skipped for that namespace; audit can note the bypass policy. |

Secret Guard applies to all content-writing paths, including REST and MCP ingest/remember/update/replace flows.

### Audit

Audit is always active and cannot be disabled in v0.3.

Audit records:

- action
- memory ID when applicable
- namespace
- caller type (`rest`, `mcp`, `cli`, `sleep`, `decay`, `system`)
- caller identity hash prefix when available
- result (`success`, `blocked`, `denied`, `warning`)
- details JSON without full request body or secret material

Audit is readable through REST and MCP, but not deletable through either.

### Startup Security Check

Startup warnings are prominent but do not block startup:

- host is `0.0.0.0`
- no API key is configured
- DB / vault / config are group- or world-readable
- CORS is broader than localhost-safe defaults
- first run suggests generating an API key

The status endpoint should expose enough security-state truth for humans and agents to notice unsafe mode.

## 6. Trust extensions

### Memory trust fields

Additive memory fields:

- `ingested_by`
- `caller_identity`
- `secret_guard_result`
- `confirmation_count`

Existing memories remain valid. Legacy rows may expose `legacy` or `null` trust values.

### Explain+

Explain+ extends existing explain output. It must preserve v0.2 provenance and add a Trust block:

- ingested by
- caller identity hash prefix or `unauthenticated`
- Secret Guard result
- confirmation count
- forgotten / replaced state
- concise audit trail

Explain+ answers the human question: "Why does Mnémlet know this, and can I trust it?"

### Forget / Replace / Confirm trust extensions

- Forget writes audit entries and records policy context.
- Replace links old and new memories in audit details.
- Confirm increments `confirmation_count` and writes confirm history.
- Existing v0.2 semantic behavior remains intact.

## 7. Namespace Policies

Namespace Policies are organizational guardrails, not RBAC.

Required policies:

| Policy | Default | Behavior |
|---|---|---|
| `secret_guard_action` | `block` | `block`, `warn`, or `allow` for that namespace. |
| `allow_recall` | `true` | Soft warning if recall is discouraged. |
| `allow_ingest` | `true` | Soft warning if ingest is discouraged. |
| `confirm_before_forget` | `false` | First forget returns `409` unless `confirm=true`. |
| `max_memories` | `0` | Soft warning if namespace exceeds the configured count; `0` means unlimited. |

Policy violations write audit warnings. The only hard blocks are Auth denial, Secret Guard `block`, and `confirm_before_forget` until confirmation is supplied.

## 8. Backup / Restore

### Backup

CLI:

```bash
mnemlet backup [--output PATH]
```

Default output:

```text
~/.mnemlet/backups/mnemlet_YYYYMMDD_HHMMSS.tar.gz
```

Snapshot contents:

- SQLite DB, including audit log.
- ChromaDB directory.
- Markdown vault.
- Config file with API key removed.

### Restore

CLI:

```bash
mnemlet restore --input PATH
```

Restore behavior:

1. Refuse unsafe restore without explicit confirmation when target data is non-empty.
2. Create a pre-restore backup of current data.
3. Replace DB, ChromaDB, and Vault as atomically as practical on local filesystem.
4. Preserve existing API key configuration unless user explicitly changes it.
5. Write audit entry for restore.
6. Run or report Startup Security Check after restore.

Backup / Restore must be tested only against isolated temporary data directories, never Christoph's live vault.

## 9. Data model

All migrations are additive.

```sql
CREATE TABLE IF NOT EXISTS audit_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT NOT NULL DEFAULT (datetime('now')),
    action TEXT NOT NULL,
    memory_id TEXT,
    namespace TEXT NOT NULL,
    caller TEXT NOT NULL,
    caller_identity TEXT,
    result TEXT DEFAULT 'success',
    details_json TEXT DEFAULT '{}'
);

CREATE INDEX IF NOT EXISTS idx_audit_action ON audit_log(action);
CREATE INDEX IF NOT EXISTS idx_audit_namespace ON audit_log(namespace);
CREATE INDEX IF NOT EXISTS idx_audit_timestamp ON audit_log(timestamp);

ALTER TABLE memories ADD COLUMN ingested_by TEXT DEFAULT NULL;
ALTER TABLE memories ADD COLUMN caller_identity TEXT DEFAULT NULL;
ALTER TABLE memories ADD COLUMN secret_guard_result TEXT DEFAULT NULL;
ALTER TABLE memories ADD COLUMN confirmation_count INTEGER DEFAULT 0;

CREATE TABLE IF NOT EXISTS namespace_policies (
    namespace TEXT NOT NULL,
    policy_key TEXT NOT NULL,
    policy_value TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    PRIMARY KEY (namespace, policy_key)
);
```

Implementation must use idempotent migration helpers. `ALTER TABLE ADD COLUMN` needs existing-column guards because SQLite does not support `IF NOT EXISTS` for that operation in all supported versions.

## 10. Multi-session execution model

v0.3 must be implementable across multiple sessions with usage limits and context-window resets.

Rules:

- One master design document: this file.
- One detailed implementation plan under `docs/superpowers/plans/`.
- Tasks are split into reviewable phases.
- Every task contains full context, files, TDD steps, verification commands, expected outputs, commit message, review gates, and rollback notes.
- Session handoff notes go under `docs/superpowers/notes/` after each phase or interruption.
- Handoffs include latest commits, test status, live daemon status, open tasks, hard constraints, and next-session start prompt.
- Fresh subagent per task.
- Subagents receive complete task text inline and must not rely on chat history.
- Review order is always Spec Review first, then Code / Quality Review.
- No task is complete without fresh verification evidence.

Live daemon rules:

- No productive POSTs to `127.0.0.1:4050` during tests.
- Demo and integration POSTs use `MNEMLET_DATA_DIR=$(mktemp -d)` and non-productive ports.
- `mnemlet.service` restarts require Christoph's explicit go immediately before restart.
- OpenWebUI is not restarted, killed, or migrated as part of v0.3 implementation.

## 12. Acceptance criteria

1. With no API key configured, existing local clients continue to work and status/startup warn clearly.
2. With an API key configured, REST and MCP requests without a valid key fail with `401`.
3. API keys are never logged, returned, stored in audit details, or included in backups.
4. Secret Guard blocks obvious GitHub, OpenAI, AWS, generic token, and password-assignment patterns by default.
5. Secret Guard warning mode records the warning and allows the write.
6. Audit records successful, denied, blocked, warning, and internal/system actions.
7. Audit query filters work for namespace, action, since, and limit.
8. Explain+ includes trust fields and a concise audit trail.
9. Namespace Policies produce warnings for soft violations and a `409` confirmation flow for protected forget operations.
10. Backup produces a restorable tarball without API key material.
11. Restore preserves or reuses current API key configuration and writes a restore audit entry.
12. README, Website, Demo, SECURITY.md, and CONTRIBUTING.md match v0.3 behavior.
13. Final full test suite passes with `.venv/bin/pytest -q`.
14. No tests or demos write to Christoph's live daemon data except the explicit final deploy step.

## 13. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Scope is large. | Phase-gated implementation, handoffs, and review after each phase. |
| Auth breaks OpenWebUI/OpenCode. | Backward-compatible no-key mode; docs and config examples before enabling key on live daemon. |
| Audit log leaks sensitive content. | Store metadata and pattern types only, never full bodies or keys. |
| Secret Guard false positives. | Conservative defaults, namespace `warn` and `allow`, targeted tests. |
| Backup / Restore damages live data. | Isolated tests only; final live operations require explicit confirmation; pre-restore backup required. |
| SQLite migration edge cases. | Additive migrations with existing-column guards and migration tests against old schemas. |

---

## 14. Rollback principles

- Before live deploy: revert commits by phase.
- After live deploy: restart `mnemlet.service` only with Christoph's explicit go.
- Auth configuration can be disabled by removing API key configuration and restarting Mnémlet.
- Secret Guard can be relaxed per namespace, but default remains `block`.
- Backup / Restore changes must never be tested against live data without explicit confirmation.
- Tags and public publishing are never moved/performed without explicit confirmation.
